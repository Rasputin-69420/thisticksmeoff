# Data Pipeline Outline for This Ticks Me Off

## Current Problems (as of now)
- The live site depends on a manually maintained data/3state_cases.json
- Adding new years or states requires manual editing
- Purple tick data is partially hardcoded in JS
- No clear, repeatable process for contributors

## Proposed New Architecture

### 1. Data Layers (clear separation)

- **Raw Layer** (data-pipeline/raw/)
  - Original CSVs from CDC, state DPH, etc.
  - Never edited by hand after import
  - Versioned with source + access date

- **Processed Layer** (data-pipeline/processed/)
  - Cleaned and normalized data
  - One file per disease per year or a structured database

- **Generated Layer** (what the site actually uses)
  - data/3state_cases.json ← generated by script
  - Should be in .gitignore (or clearly marked as generated)

### 2. Pipeline Tools

Recommended stack (simple but powerful):
- Python (pandas + openpyxl) — best for CSV/Excel work
- One main script: uild_data.py
- Optional: alidate_data.py

### 3. How Adding New Data Works (Target Experience)

Example workflow for adding 2026 data:

1. Drop new CSV into data-pipeline/raw/2026/
2. Run: python data-pipeline/scripts/build_data.py
3. Script:
   - Validates the new data
   - Applies deduplication rules
   - Updates the processed layer
   - Regenerates data/3state_cases.json
4. Test the site in website-testnet/
5. Commit the raw data + any script changes

### 4. Purple Tick Data

- Move all purple data into structured CSVs/JSON under data-pipeline/raw/purple/
- Include year, state, estimated_counties, source, notes
- Script generates the establishedCounts object the site needs

### 5. Benefits

- Much easier to add new years/states
- Full audit trail of data changes
- Can support county-level data later without breaking the site
- Multiple people can contribute data without breaking the visualization
- Clear separation between "data work" and "visualization work"
